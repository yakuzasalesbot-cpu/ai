# Handoff — ЯКУДЗА

**Дата фіксації:** 2026-04-28

## 1. Головна ціль

Поточна ціль не в тому, щоб вигадати нову систему, а в тому, щоб довести існуючу Google-таблицю `ЯКУДЗА` до стабільного робочого стану.

Кінцевий результат v1:
- оператор працює у 10-денних картках техніки і генератора;
- `Реєстр`, `Залишки`, `Донесення` наповнюються з тих самих даних;
- після перевірки можна зробити фінальний Excel для друку;
- не треба вручну по кілька разів переписувати дорожні листи через арифметичні помилки.

## 2. Джерело істини

Головне джерело істини:
- Google-таблиця `Дорожні листи та генератори`
- Sheet ID: `1SXjTO2cc7gzeAA_mjfAhUIZlX1o1ZpdBJAoM5bR5fDw`

Локальні `.xlsx`, скрипти та експорти:
- допоміжні;
- не є головним контекстом;
- використовуються для перевірки або фінального друку.

## 3. Поточна логіка роботи

1. По кожній машині є 3 блоки по 10 днів.
2. У кожному блоці є маршрут, пробіг, пальне, залишки, спідометр.
3. Для генератора є окремий агрегатний блок з мотогодинами і пальним.
4. Верх кожного блока має містити:
   - `Дата документа`
   - значення дати
   - `№ документа`
   - значення номера
5. На базі цих блоків формуються:
   - `Реєстр`
   - `Залишки`
   - `Донесення`

## 4. Що вже зроблено

- створено `PROJECT_CONTEXT.md` як головний короткий контекст;
- створено `AI_TASK_LOG.md` як журнал ітерацій;

- у живій Google-таблиці були дозаповнені верхні поля `Дата документа` і `№ документа` в активних блоках;
- частина переносів стартового залишку між блоками була виправлена прямо в Google-таблиці;
- локальні скрипти перевірки й генерації вже існують, але поки не є головним робочим контуром.

## 5. Що ще не закрито

- треба довести всі активні вкладки до одного жорсткого шаблону;
- треба окремо перевірити логіку по `КОЗАК 4277`, якщо там лишився реальний мінус по паливу;
- треба перевірити, чи `Реєстр`, `Залишки`, `Донесення` після останніх правок тягнуть правильні значення;
- треба розвести стару документацію і нову, щоб не було двох паралельних реальностей.

## 6. Правила для наступної LLM

- не домислювати цифри;
- не виправляти облікові дані без підтвердження, якщо це не однозначний перенос;
- не починати велику задачу без читання:
  - `docs/01_AI_WORK_RULES.md`
  - `docs/02_MULTI_LLM_PROTOCOL.md`
  - `docs/03_API_LIMIT_GUIDELINES.md`
  - `docs/04_ITERATION_PROTOCOL.md`
  - `docs/05_HANDOFF_PROMPTS.md`
- перед роботою прочитати:
  - `PROJECT_CONTEXT.md`
  - `AI_TASK_LOG.md`
  - `СТАН_ПРОЕКТУ.md`
- після ітерації залишити короткий запис у `AI_TASK_LOG.md`.

## 7. Пріоритетний порядок дій

1. Уніфікувати всі робочі вкладки техніки.
2. Уніфікувати генераторний блок.
3. Перевірити переходи:
   - спідометр
   - мотогодини
   - залишок пального
4. Перевірити фінальні вкладки:
   - `Реєстр`
   - `Залишки`
   - `Донесення`
5. Лише після цього повертатись до повного автоматичного експорту Excel.

## 8. Якщо починати нову сесію

Початок має бути таким:

1. Прочитати `PROJECT_CONTEXT.md`.
2. Прочитати `AI_TASK_LOG.md`.
